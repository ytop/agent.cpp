# Requirements Document

## Introduction

This feature is a clean, native C++ re-implementation of the existing TypeScript application that lives in `coding-agent/` (the npm package `@earendil-works/pi-coding-agent`, binary name `pi`). The new implementation lives in a separate top-level directory `pie-coding-agent/` and MUST NOT modify any file under `coding-agent/`. The C++ port is compiled with Apple clang version 17.0.0 (C++20, macOS, Apple Silicon as primary target and Intel x86_64 as secondary) and produces a single `pie` binary plus an embeddable C++ SDK library.

The port reaches feature parity with the TS application across modes (interactive TUI, print, JSON event stream, RPC, export, SDK), built-in tools, sessions and tree branching, compaction, the full provider/model system, authentication, configuration and settings, context files, customization subsystems (prompt templates, skills, themes, extensions), Pie packages, the interactive UI (editor, footer, header, modals, message rendering, image rendering), slash commands, keybindings, message queue semantics (steering vs follow-up), HTTP export and sharing, telemetry and update checks, image handling (drag/drop, paste, auto-resize, EXIF, inline rendering), bash execution semantics, and v1/v2 → v3 session migration. The port reads and writes the same on-disk JSON/JSONL formats so existing data under `~/.pi/agent/` continues to work seamlessly when switched between the TS and C++ binaries.

This document specifies the requirements at the level of testable behavior, including correctness properties suitable for property-based testing where applicable. Implementation choices (CMake, dependencies, library selection, extension runtime mechanism) are decided in the Design phase and constrained where the User explicitly requested options.

## Glossary

- **Pie_Cpp**: The new C++ application produced by this spec. Single binary called `pie`, built from `pie-coding-agent/`.
- **Pi_TS**: The existing TypeScript application in `coding-agent/`. NOT modified by this spec; serves as the reference implementation and wire-format definition.
- **TS_Wire_Format**: The exact byte/JSON layout produced by Pi_TS for any persisted artifact (sessions, settings, auth, themes, models registry, keybindings, gist payloads, JSON-stream events, RPC commands and responses). Pie_Cpp MUST round-trip read/write this format byte-for-byte where the format is canonical, and semantically equivalently where the format permits insignificant whitespace.
- **Pie_Cpp_Binary**: The compiled `pie` binary produced by the C++ build.
- **Pie_Cpp_SDK**: The C++ static or shared library form of Pie_Cpp, embeddable in third-party C++ programs, exposing equivalents of `createAgentSession` and `createAgentSessionRuntime`.
- **Agent_Dir**: The directory containing global config, default `~/.pie/agent`. Overridable via `PIE_CODING_AGENT_DIR`.
- **Session_Dir**: The directory containing session JSONL files, default `<Agent_Dir>/sessions`. Overridable via `--session-dir`, `PIE_CODING_AGENT_SESSION_DIR`, or `sessionDir` setting.
- **Session_File**: A JSONL file under `<Session_Dir>/--<cwd-with-slashes-replaced>--/<timestamp>_<uuid>.jsonl`.
- **Session_Entry**: A single JSON object on one line of a Session_File. Has a `type` field (`session`, `message`, `compaction`, `branch_summary`, `model_change`, `thinking_level_change`, `custom`, `custom_message`, `label`, `session_info`).
- **Session_Header**: The first entry in a Session_File, with `type: "session"` and a `version` field.
- **Session_Tree**: The directed tree formed by Session_Entries, where every non-header entry has a unique 8-character hex `id` and a `parentId` (or `null` for the first entry).
- **Session_Leaf**: The current position in the Session_Tree; the entry whose path to the root forms the active conversation.
- **Branch_Summary**: An LLM-generated summary entry inserted when the user navigates to a different branch via `/tree`, used to seed the new branch with context from the abandoned path.
- **Compaction**: The operation of summarizing earlier messages in the active path to free context, replacing them with a `compaction` entry that records `summary`, `firstKeptEntryId`, and `tokensBefore`.
- **Build_Session_Context**: The deterministic walk from Session_Leaf to Session_Header that produces the message list submitted to the LLM. Defined in `coding-agent/docs/session-format.md`.
- **Thinking_Level**: One of `off`, `minimal`, `low`, `medium`, `high`, `xhigh`. Controls reasoning token budgets for models that support them.
- **Provider**: A configured LLM service (Anthropic, OpenAI, Google, Bedrock, etc.) plus authentication (subscription OAuth or API key).
- **Model**: A specific model identified by `(provider, modelId)` available via a Provider.
- **Model_Registry**: The merged registry of built-in tool-capable models plus custom entries from `<Agent_Dir>/models.json`.
- **Auth_Storage**: The auth file at `<Agent_Dir>/auth.json` storing OAuth tokens and API keys.
- **Settings_Layer**: One of (defaults, global `<Agent_Dir>/settings.json`, project `.pie/settings.json`, runtime overrides). Effective settings are the deep merge in that precedence order.
- **Builtin_Tool**: One of `read`, `write`, `edit`, `bash`, `grep`, `find`, `ls`.
- **Tool_Allowlist**: The set of tool names enabled, derived from `--tools` / `-t`, `--no-builtin-tools` / `-nbt`, and `--no-tools` / `-nt`.
- **Steering_Message**: A queued message delivered after the current assistant turn finishes its tool calls.
- **Follow_Up_Message**: A queued message delivered only after the agent finishes all work.
- **Message_Queue_Mode**: `"one-at-a-time"` (default) or `"all"`, configurable per queue (steering, follow-up).
- **Extension_Model**: The mechanism Pie_Cpp uses to support pluggable extensions equivalent in capability surface to the Pi_TS `ExtensionAPI`. The exact mechanism (embedded JS runtime, native shared-object plugins, or out-of-process JSON-RPC) is decided in Design.
- **Pie_Package**: A bundle of extensions, skills, prompt templates, and themes installable from npm, git, https, or ssh sources via `pie install`.
- **Skill**: A package containing a `SKILL.md` providing on-demand capability instructions, discoverable by name and invokable as `/skill:<name>`.
- **Prompt_Template**: A markdown file with optional frontmatter and `{{variable}}` substitutions, expanded when typed as `/<name>`.
- **Theme**: A JSON file describing TUI colors, hot-reloaded on file change.
- **Context_File**: An `AGENTS.md` or `CLAUDE.md` file loaded at startup and concatenated into the system context.
- **Print_Mode**: Non-interactive single-response mode invoked with `-p` / `--print`.
- **Json_Mode**: Event-stream mode invoked with `--mode json` that emits all session events as JSONL on stdout.
- **Rpc_Mode**: Bidirectional mode invoked with `--mode rpc`, using strict LF-delimited JSONL on stdin/stdout per `coding-agent/docs/rpc.md`.
- **Export_Mode**: One-shot HTML export invoked with `--export <in> [out]`.
- **Round_Trip_Property**: For any encoder `E` and decoder `D`, the property `D(E(x))` is semantically equal to `x` for all valid `x` in the encoder's domain.
- **Structurally_Equal**: Two parsed JSON values are structurally equal if and only if their decoded JSON values per RFC 8259 are recursively equal, treating JSON objects as unordered maps, JSON arrays as ordered sequences, JSON numbers by their decoded numeric value, and JSON strings by their Unicode codepoint sequence.

## Requirements

### Requirement 1: Project Layout, Build, and Distribution

**User Story:** As a developer building Pie_Cpp from source on macOS, I want a single CMake-driven build that compiles cleanly with Apple clang 17.0.0, so that I can produce a working `pie` binary and an embeddable SDK library out of the box.

#### Acceptance Criteria

1. THE Pie_Cpp source SHALL live entirely under `pie-coding-agent/` at the repository root.
2. THE Pie_Cpp build SHALL NOT modify, read for build inputs, or depend on any file under `coding-agent/` at build time.
3. WHEN a developer runs the build command published in `pie-coding-agent/README.md` on macOS 13 or later with Apple clang version 17.0.0 on `PATH`, THE Pie_Cpp build SHALL produce an executable file at `pie-coding-agent/build/pie` in a single invocation, with no manual editing of source files required, and that executable SHALL exit with status 0 within 5 seconds when invoked as `pie --version` and SHALL print a non-empty version string to stdout.
4. THE Pie_Cpp build SHALL compile all Pie_Cpp and Pie_Cpp_SDK translation units with the C++20 standard.
5. THE Pie_Cpp build SHALL target macOS arm64 (Apple Silicon) as the primary architecture and macOS x86_64 as the secondary architecture.
6. THE Pie_Cpp build SHALL pin every third-party dependency to a single immutable version identifier (a release tag, exact semantic version, or full commit SHA) declared in exactly one manifest file (vcpkg.json, conanfile, or git submodule pins) checked into `pie-coding-agent/`, and the manifest SHALL NOT contain version ranges, caret or tilde specifiers, floating branch names, or `latest` references.
7. THE Pie_Cpp build SHALL produce, in addition to the `pie` executable from criterion 3, a Pie_Cpp_SDK library artifact (static or shared library file) at a documented path under `pie-coding-agent/build/` together with a public header directory under `pie-coding-agent/include/` that another C++ program can link against and `#include` to construct an agent session.
8. WHEN a developer invokes `clang++ -std=c++20 -O2` against the documented Pie_Cpp_SDK entry-point header and dependencies on macOS with Apple clang 17.0.0, THE Pie_Cpp source SHALL compile and link without errors or warnings escalated to errors.
9. WHEN the `pie` executable produced by criterion 3 is invoked as `pie --version` on a host where no `node`, `npm`, `bun`, or `deno` executable is present on `PATH`, THE Pie_Cpp_Binary SHALL exit with status 0 and SHALL NOT spawn any of those runtimes.
10. THE Pie_Cpp build documentation SHALL explicitly enumerate macOS 13 or later (arm64 and x86_64) and Linux (x86_64 and aarch64) as the only supported host platforms, SHALL declare Windows as out of scope, and SHALL NOT claim Windows support.
11. IF the build is invoked on macOS with an Apple clang version other than 17.0.0 detected on `PATH`, THEN THE Pie_Cpp build SHALL halt before compiling any source file, SHALL emit a diagnostic identifying the detected clang version and the required version, and SHALL exit with a non-zero status without producing a `pie` executable or Pie_Cpp_SDK library artifact.

### Requirement 2: Command-Line Interface and Mode Selection

**User Story:** As a user invoking `pie` from the shell, I want the same CLI surface as Pi_TS, so that my existing scripts, aliases, and muscle memory continue to work.

#### Acceptance Criteria

1. THE Pie_Cpp_Binary SHALL accept the same top-level argument grammar as Pi_TS, namely `[options] [@files...] [messages...]`, where positional tokens beginning with `@` are treated as file inclusions and all remaining positional tokens are concatenated in argument order with a single space separator to form the initial prompt.
2. WHEN invoked with no arguments and both stdin and stdout are TTYs, THE Pie_Cpp_Binary SHALL enter interactive TUI mode.
3. WHEN invoked with `-p` or `--print` and the agent completes its response without error, THE Pie_Cpp_Binary SHALL execute Print_Mode by sending the prompt, streaming the assistant response text to stdout (and only the response text, with no progress chrome, prompts, or status lines on stdout), and exiting with exit code 0.
4. IF Print_Mode fails before or during response streaming due to authentication, network, model-provider, or input-validation errors, THEN THE Pie_Cpp_Binary SHALL write a diagnostic to stderr identifying the failure category and SHALL exit with a non-zero exit code distinct from the exit code used for argument errors.
5. WHEN invoked in Print_Mode and stdin is not a TTY (i.e., stdin is a pipe, file, or other non-terminal stream), THE Pie_Cpp_Binary SHALL read all bytes from stdin until EOF, decode them as UTF-8, and prepend them to the initial prompt formed from positional message tokens, separated from the message tokens by a single `\n` character, matching Pi_TS behavior.
6. WHEN invoked with `--mode json`, THE Pie_Cpp_Binary SHALL execute Json_Mode and emit the same set of event types defined in `coding-agent/docs/json.md` as JSONL on stdout, with the Session_Header as the first line, exactly one JSON object per line terminated by `\n`, and SHALL write any human-readable diagnostics to stderr only and never to stdout.
7. WHEN invoked with `--mode rpc`, THE Pie_Cpp_Binary SHALL execute Rpc_Mode and implement the protocol defined in `coding-agent/docs/rpc.md`, including strict LF-only record framing on both stdin and stdout, optional `\r\n` tolerance on input by stripping a trailing `\r`, and rejection of generic line readers' Unicode separators (records SHALL NOT be split on `U+2028` or `U+2029`).
8. WHEN invoked with `--export <in> [out]`, THE Pie_Cpp_Binary SHALL render the JSONL session at `<in>` to a self-contained HTML file using the export-html template, write it to `<out>` when provided or to the default output path used by Pi_TS's `--export` when `<out>` is omitted, and exit with exit code 0 on success.
9. IF more than one mode-selecting flag from the set {`-p` or `--print`, `--mode json`, `--mode rpc`, `--export`} is supplied in a single invocation, THEN THE Pie_Cpp_Binary SHALL write a diagnostic to stderr naming the conflicting flags and SHALL exit with a non-zero exit code without entering any mode.
10. THE Pie_Cpp_Binary SHALL accept all model selection flags from Pi_TS: `--provider`, `--model` (including `provider/id` and `provider/id:thinking` shorthands), `--api-key`, `--thinking`, `--models`, `--list-models [search]`.
11. THE Pie_Cpp_Binary SHALL accept all session flags from Pi_TS: `-c` / `--continue`, `-r` / `--resume`, `--session <path|id>`, `--fork <path|id>`, `--session-dir <dir>`, `--no-session`.
12. THE Pie_Cpp_Binary SHALL accept all tool flags from Pi_TS: `--tools <list>` / `-t`, `--no-builtin-tools` / `-nbt`, `--no-tools` / `-nt`.
13. THE Pie_Cpp_Binary SHALL accept all resource flags from Pi_TS: `-e` / `--extension`, `--no-extensions`, `--skill`, `--no-skills`, `--prompt-template`, `--no-prompt-templates`, `--theme`, `--no-themes`, `--no-context-files` / `-nc`.
14. THE Pie_Cpp_Binary SHALL accept all other flags from Pi_TS: `--system-prompt <text>`, `--append-system-prompt <text>`, `--verbose`, `-h` / `--help`, `-v` / `--version`, `--offline`.
15. WHEN the argument list contains tokens prefixed with `@`, THE Pie_Cpp_Binary SHALL treat each `@<path>` as a file inclusion in the same manner as Pi_TS: text content inlined into the prompt, image content sent as ImageContent with base64-encoded data and a detected MIME type.
16. IF an `@<path>` token references a file that does not exist or is not readable by the current process, THEN THE Pie_Cpp_Binary SHALL write a diagnostic to stderr identifying the offending path and the failure cause, SHALL NOT send any prompt to the LLM, and SHALL exit with a non-zero exit code.
17. WHEN invoked with `pie install`, `pie remove`, `pie uninstall`, `pie update`, `pie list`, or `pie config`, THE Pie_Cpp_Binary SHALL dispatch to the package manager subsystem defined in Requirement 16.
18. IF an unrecognized flag is supplied, THEN THE Pie_Cpp_Binary SHALL write to stderr an error message identifying the unrecognized flag verbatim followed by a one-line usage hint pointing to `pie --help`, and SHALL exit with a non-zero exit code reserved for argument errors that is distinct from the exit code used for runtime or operation errors.


### Requirement 3: Session File Format and Wire Compatibility

**User Story:** As a user with existing Pi_TS sessions in `~/.pi/agent/sessions/`, I want Pie_Cpp to read and write the same session file format, so that I can switch between the TS and C++ binaries without losing or corrupting data.

#### Acceptance Criteria

1. THE Pie_Cpp session subsystem SHALL read JSONL Session_Files conforming to versions 1, 2, and 3 of the format described in `coding-agent/docs/session-format.md`.
2. WHEN Pie_Cpp opens a Session_File whose Session_Header has `version` 1 or 2, THE Pie_Cpp session subsystem SHALL migrate the in-memory representation to version 3 semantics on load (legacy linear sequence becomes a tree, `hookMessage` role becomes `custom`) per the migration rules in Requirement 23.
3. THE Pie_Cpp session subsystem SHALL write Session_Files whose first line is a Session_Header JSON object containing exactly the fields `type` (string `"session"`), `version` (integer `3`), `id` (string), `timestamp` (string), and `cwd` (string), plus the optional `parentSession` field (string) only when the session was created from a parent, with field types and shapes matching `coding-agent/docs/session-format.md` and no additional top-level keys beyond those documented there.
4. THE Pie_Cpp session subsystem SHALL encode every Session_File as UTF-8 with no byte-order mark, emit each non-header Session_Entry as a single JSON object on its own line terminated by exactly one `\n` (`U+000A`) byte, contain no `\r`, no other whitespace bytes between the closing `}` of an entry and its terminating `\n`, and no blank lines between entries.
5. THE Pie_Cpp session subsystem SHALL preserve all entry types from `coding-agent/docs/session-format.md`: `session`, `message`, `compaction`, `branch_summary`, `model_change`, `thinking_level_change`, `custom`, `custom_message`, `label`, `session_info`.
6. THE Pie_Cpp session subsystem SHALL preserve all message roles: `user`, `assistant`, `toolResult`, `bashExecution`, `custom`, `branchSummary`, `compactionSummary`.
7. THE Pie_Cpp session subsystem SHALL preserve all content block types: `text`, `image` (with base64 `data` and `mimeType`), `thinking`, `toolCall`.
8. THE Pie_Cpp session subsystem SHALL preserve all `Usage` fields including `input`, `output`, `cacheRead`, `cacheWrite`, `totalTokens`, and the nested `cost` object with sub-fields `input`, `output`, `cacheRead`, `cacheWrite`, `total`.
9. THE Pie_Cpp session subsystem SHALL generate `id` values for every non-header Session_Entry that match the regular expression `^[0-9a-f]{8}$` and SHALL ensure each generated `id` is unique among all non-header entries already present in the same Session_File.
10. WHEN Pie_Cpp writes any new entry to a Session_File, THE Pie_Cpp session subsystem SHALL acquire an exclusive file lock on the Session_File for the duration of the append using the on-disk convention used by `proper-lockfile`: a sibling lockfile path of the form `<session-file>.lock`, retry acquisition at intervals not exceeding 100 ms for up to 10 s before failing, treat any existing lockfile older than 10 s as stale and reclaim it, and release the lock immediately after the append completes or on process exit.
11. **Round-trip property:** FOR ALL valid Session_File byte streams `S` produced by Pi_TS, the operation `parse(S)` followed by `serialize(parse(S))` SHALL produce a byte stream `S'` such that `parse(S')` is Structurally_Equal to `parse(S)`.
12. **Round-trip property:** FOR ALL Session_Tree values `T` constructible via the SessionManager API, the operation `serialize(T)` followed by `parse(serialize(T))` SHALL yield a Session_Tree Structurally_Equal to `T`, with identical entry `id` values, identical `parentId` links, and identical leaf position.
13. **Pretty printer requirement:** THE Pie_Cpp session subsystem SHALL provide a deterministic pretty printer that converts a Session_Entry value to a canonical JSONL line whose canonical form is: UTF-8 with no byte-order mark, a single JSON object containing no insignificant whitespace between tokens, key order matching the order documented for that entry type in `coding-agent/docs/session-format.md`, and a single terminating `\n`; for the same input value, the printer SHALL produce byte-identical output across processes and runs, and SHALL be the inverse of the Session_Entry parser per the round-trip property in criterion 11.
14. IF a Session_File contains a malformed line that is not valid JSON, THEN THE Pie_Cpp session subsystem SHALL emit a diagnostic to stderr containing the absolute file path, the 1-based line number of the malformed line, and a brief reason; SHALL skip the malformed line; SHALL continue parsing remaining lines; and SHALL NOT modify the Session_File on disk as a side effect of the parse error.
15. WHEN Pie_Cpp's append to a Session_File is interrupted before completion (process crash, signal, or power loss), THE Pie_Cpp session subsystem SHALL ensure that the Session_File on disk ends at a `\n`-terminated complete line and contains no partially-written entry, such that a subsequent `parse` of the file by either Pie_Cpp or Pi_TS succeeds without invoking the malformed-line recovery path in criterion 14.

### Requirement 4: SessionManager API and Tree Navigation

**User Story:** As an SDK consumer or extension author, I want the same SessionManager surface in C++ that Pi_TS exposes, so that I can manipulate sessions, walk the tree, branch, and label entries programmatically.

#### Acceptance Criteria

1. THE Pie_Cpp_SDK SHALL provide a `SessionManager` type with the static factories `create(cwd, sessionDir?)`, `open(path, sessionDir?)`, `continueRecent(cwd, sessionDir?)`, `inMemory(cwd?)`, and `forkFrom(sourcePath, targetCwd, sessionDir?)`.
2. THE Pie_Cpp_SDK SHALL provide static listing methods `list(cwd, sessionDir?, onProgress?)` and `listAll(onProgress?)`.
3. THE Pie_Cpp_SDK `SessionManager` SHALL expose the appending methods `appendMessage`, `appendThinkingLevelChange`, `appendModelChange`, `appendCompaction(summary, firstKeptEntryId, tokensBefore, details?, fromHook?)`, `appendCustomEntry(customType, data?)`, `appendSessionInfo(name)`, `appendCustomMessageEntry(customType, content, display, details?)`, and `appendLabelChange(targetId, label)`; each appending method SHALL return the 8-character lowercase hex `id` of the newly-created Session_Entry, set that entry's `parentId` to the current Session_Leaf id (or `null` if this is the first non-header entry), and advance the Session_Leaf to the new entry.
4. THE Pie_Cpp_SDK `SessionManager` SHALL expose tree-navigation methods `getLeafId` (returns `null` before any non-header entry is appended), `getLeafEntry`, `getEntry(id)`, `getBranch(fromId?)` (defaulting to the current Session_Leaf when `fromId` is omitted), `getTree`, `getChildren(parentId)`, `getLabel(id)`, `branch(entryId)`, `resetLeaf`, and `branchWithSummary(entryId, summary, details?, fromHook?)`; `getEntry`, `getLabel`, and `getChildren` SHALL return a defined empty/absent value when their argument is not present in the Session_Tree, and `branch` and `branchWithSummary` SHALL reject `entryId` values not present in the Session_Tree.
5. THE Pie_Cpp_SDK `SessionManager` SHALL expose context and info methods: `buildSessionContext`, `getEntries`, `getHeader`, `getSessionName`, `getCwd`, `getSessionDir`, `getSessionId`, `getSessionFile`, `isPersisted`.
6. THE Pie_Cpp_SDK `SessionManager` SHALL expose session-management methods: `newSession(options?)` (with optional `parentSession`), `setSessionFile(path)`, `createBranchedSession(leafId)`.
7. WHEN `buildSessionContext` is invoked with a Session_Leaf `L`, THE Pie_Cpp SHALL produce the message list following the algorithm in `coding-agent/docs/session-format.md`:
   1. Walk from `L` toward the root collecting entries on the path in deterministic root-to-leaf order.
   2. Extract the current Model and Thinking_Level from the most recent `model_change` and `thinking_level_change` entries on the path.
   3. IF exactly one `compaction` entry `C` is on the path, THEN emit `C.summary` first, then messages from `C.firstKeptEntryId` to `C` in path order, then messages after `C` in path order.
   4. Convert `branch_summary` and `custom_message` entries to the appropriate message format defined in `coding-agent/docs/session-format.md`.
8. **Property:** FOR ALL Session_Trees `T` and all valid leaf ids `L` in `T`, `buildSessionContext(T, L)` SHALL return the same message list regardless of the order in which the entries were appended, provided the parent-child relations are unchanged.
9. **Property:** FOR ALL Session_Trees `T`, the result of evaluating `getBranch(getLeafId())` SHALL contain only entries whose `id` appears in the result of `getEntries()`, and the path SHALL be acyclic from leaf to header.
10. WHEN `branch(entryId)` is called with an `entryId` present in the Session_Tree, THE Pie_Cpp SessionManager SHALL move the in-memory Session_Leaf pointer to `entryId` and SHALL NOT delete or modify any pre-existing entry.
11. WHEN `branchWithSummary(entryId, summary)` is called with an `entryId` present in the Session_Tree, THE Pie_Cpp SessionManager SHALL append a `branch_summary` entry whose `parentId` is `entryId` and whose `fromId` is the prior Session_Leaf id, and SHALL move the Session_Leaf to the new `branch_summary` entry.
12. WHEN `resetLeaf` is called, THE Pie_Cpp SessionManager SHALL set the Session_Leaf to `null` and `buildSessionContext` invoked while the Session_Leaf is `null` SHALL return an empty message list with the model and Thinking_Level from the Session_Header (or defaults if not set).
13. **Invariant:** FOR ALL appended Session_Entries, the entry's `parentId` SHALL reference an `id` already present in the Session_Tree, except for the very first non-header entry whose `parentId` SHALL be `null`.

### Requirement 5: Settings, Configuration, and Environment Variables

**User Story:** As a user with `~/.pi/agent/settings.json` and project-level `.pie/settings.json`, I want Pie_Cpp to merge settings using the same deep-merge semantics as Pi_TS, so that my configuration applies identically.

#### Acceptance Criteria

1. WHEN `<Agent_Dir>/settings.json` exists and is readable, THE Pie_Cpp settings subsystem SHALL load the global settings layer from that file; otherwise the global settings layer SHALL be treated as the empty JSON object `{}`.
2. WHEN `<cwd>/.pie/settings.json` exists and is readable, THE Pie_Cpp settings subsystem SHALL load the project settings layer from that file; otherwise the project settings layer SHALL be treated as the empty JSON object `{}`.
3. IF a settings file at `<Agent_Dir>/settings.json` or `<cwd>/.pie/settings.json` exists but cannot be parsed as a JSON object or cannot be read due to an I/O error, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic to stderr identifying the file path and a one-line reason, treat the offending layer as the empty JSON object `{}`, and continue startup.
4. WHEN both global and project settings are loaded, THE Pie_Cpp settings subsystem SHALL deep-merge nested JSON objects with project values overriding global values key-by-key, where any non-object value (including arrays, strings, numbers, booleans, and null) at a given path SHALL be replaced wholesale rather than merged element-by-element.
5. THE Pie_Cpp settings subsystem SHALL recognize every setting documented in `coding-agent/docs/settings.md`, including categories `Model & Thinking`, `UI & Display`, `Warnings`, `Compaction`, `Branch Summary`, `Retry`, `Message Delivery`, `Terminal & Images`, `Shell`, `Sessions`, `Model Cycling`, `Markdown`, `Resources`.
6. THE Pie_Cpp settings subsystem SHALL apply the defaults documented in `coding-agent/docs/settings.md` when a setting is unspecified by both global and project layers.
7. THE Pie_Cpp settings subsystem SHALL honor the precedence `--session-dir` > `PIE_CODING_AGENT_SESSION_DIR` > `sessionDir` setting > default for the session directory.
8. THE Pie_Cpp settings subsystem SHALL recognize and apply the following environment variables: `PIE_CODING_AGENT_DIR`, `PIE_CODING_AGENT_SESSION_DIR`, `PIE_PACKAGE_DIR`, `PIE_OFFLINE`, `PIE_SKIP_VERSION_CHECK`, `PIE_TELEMETRY`, `PIE_CACHE_RETENTION`, `VISUAL`, `EDITOR`, `PIE_SHARE_VIEWER_URL`.
9. **Truthy/falsy environment variable values:** FOR boolean-style environment variables (`PIE_OFFLINE`, `PIE_SKIP_VERSION_CHECK`, `PIE_TELEMETRY`), THE Pie_Cpp_Binary SHALL treat the case-insensitive values `1`, `true`, `yes`, `on` as truthy and SHALL treat the case-insensitive values `0`, `false`, `no`, `off`, and the empty string as falsy.
10. WHEN `PIE_OFFLINE` is set to a truthy value or `--offline` is passed, THE Pie_Cpp_Binary SHALL NOT initiate any of the following network operations during startup: update checks against `https://pi.dev/api/latest-version`, package update checks, install/update telemetry to `https://pi.dev/api/report-install`.
11. WHEN `PIE_SKIP_VERSION_CHECK` is set to a truthy value, THE Pie_Cpp_Binary SHALL NOT initiate the version check against `https://pi.dev/api/latest-version` while leaving other network operations governed by other flags.
12. **Deep-merge property:** FOR ALL JSON object values `G` (global) and `P` (project), the merge `M = deep_merge(G, P)` SHALL satisfy: for every leaf path `p`, `M[p]` equals `P[p]` if `p` is set in `P`, otherwise `M[p]` equals `G[p]` if `p` is set in `G`, otherwise `M[p]` is unset.
13. **Deep-merge property:** FOR ALL JSON object values `X`, `deep_merge(X, X)` SHALL be Structurally_Equal to `X` (idempotence).
14. WHEN the `/settings` interactive command is invoked, THE Pie_Cpp interactive UI SHALL allow editing the same set of common settings exposed by Pi_TS's `/settings` UI at the matching version.
15. WHEN the user persists changes via `/settings` or via the Pie_Cpp_SDK SettingsManager, THE Pie_Cpp settings subsystem SHALL write the global settings file at `<Agent_Dir>/settings.json` using the same JSON shape Pi_TS produces; IF the write fails due to permission, disk-space, or I/O error, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic identifying the file path and reason, leave the prior on-disk file unchanged, and continue running.


### Requirement 6: Authentication, Providers, and Model Registry

### Requirement 8: Compaction Subsystem

**User Story:** As a user running long sessions, I want manual `/compact`, automatic threshold compaction, automatic overflow recovery, and `/tree` branch summarization to work the same as Pi_TS, so that my long sessions stay within the context window without losing important context.

#### Acceptance Criteria

1. WHEN `/compact` is invoked with no argument, THE Pie_Cpp compaction subsystem SHALL emit `compaction_start` with `reason: "manual"`, summarize all messages on the active path from the previous kept boundary (or Session_Header if none) up to the latest valid cut point per the cut-point rules in `coding-agent/docs/compaction.md` (a `user`, `assistant`, `bashExecution`, or `custom` message; never a `toolResult`) using the default compaction prompt, append a `compaction` Session_Entry whose fields conform to Requirement 8.5, and emit `compaction_end` with `reason: "manual"`, `aborted: false`, `willRetry: false`, and the resulting `result`.
2. WHEN `/compact <prompt>` is invoked with a non-empty `<prompt>`, THE Pie_Cpp compaction subsystem SHALL perform the same operation as Requirement 8.1 with `<prompt>` appended to the default compaction prompt as additional summarization instructions, and SHALL append a `compaction` Session_Entry whose fields conform to Requirement 8.5 without adding any extra fields.
3. WHILE `compaction.enabled` is `true` (default), WHEN, after appending any new Session_Entry to the active path, the condition `contextTokens > contextWindow - compaction.reserveTokens` (with `contextTokens` computed from `buildSessionContext` and `contextWindow` taken from the active Model's context length) becomes true, THE Pie_Cpp compaction subsystem SHALL emit `compaction_start` with `reason: "threshold"`, perform compaction so that messages newer than `compaction.keepRecentTokens` tokens (counting backward from the leaf) are kept, append a `compaction` Session_Entry conforming to Requirement 8.5, and emit `compaction_end` with `reason: "threshold"`, `aborted: false`, `willRetry: false`, and the resulting `result`.
4. IF the LLM provider returns a context-overflow error during a turn, THEN THE Pie_Cpp compaction subsystem SHALL emit `compaction_start` with `reason: "overflow"`, perform compaction, emit `compaction_end` with `reason: "overflow"`, `aborted: false`, `willRetry: true`, and the resulting `result`, and THEN automatically retry the original prompt exactly once.
5. THE Pie_Cpp compaction `compaction` Session_Entries SHALL contain the required fields `summary`, `firstKeptEntryId`, `tokensBefore`, and the optional fields `details` and `fromHook` exactly as defined in `coding-agent/docs/session-format.md`, with optional fields omitted when not set.
6. WHEN the user navigates to a different Session_Tree branch via `/tree` and `branchSummary.skipPrompt` is `false`, THE Pie_Cpp compaction subsystem SHALL prompt the user with a Yes/No confirmation to summarize the abandoned branch, and on a Yes confirmation SHALL generate a Branch_Summary whose serialized input to the summarization LLM does not exceed `branchSummary.reserveTokens` tokens and SHALL append a `branch_summary` Session_Entry whose `fromId` is the abandoned Session_Leaf id and whose remaining fields conform to `coding-agent/docs/session-format.md`.
7. IF `branchSummary.skipPrompt` is `true`, THEN THE Pie_Cpp compaction subsystem SHALL NOT display the summarization prompt and SHALL skip generation of a `branch_summary` entry on `/tree` navigation.
8. THE Pie_Cpp compaction subsystem SHALL emit `compaction_start` and `compaction_end` events in Json_Mode and Rpc_Mode using the same field set as Pi_TS: `compaction_start` carries `reason` (one of `manual`, `threshold`, `overflow`); `compaction_end` carries `reason` (same values), `aborted` (boolean), `willRetry` (boolean), `result` (object or undefined), and optional `errorMessage` (string).
9. IF compaction summarization fails (the summarization LLM call returns an error or is aborted by the user), THEN THE Pie_Cpp compaction subsystem SHALL NOT append a `compaction` Session_Entry, SHALL leave the Session_Leaf unchanged, SHALL emit `compaction_end` with `aborted: true`, `result: undefined`, `willRetry: false` (or `willRetry: true` only when the trigger was `overflow` and a retry will occur), and `errorMessage` indicating the failure cause, and SHALL surface the failure to the caller (interactive UI, Json_Mode stream, or Rpc_Mode response) without modifying any prior Session_Entry.


### Requirement 9: Message Queue (Steering and Follow-Up)

**User Story:** As a user typing messages while the agent is working, I want Enter to queue a steering message and Alt+Enter to queue a follow-up message, with the same delivery modes (`one-at-a-time` and `all`) as Pi_TS, so that I can guide and pipeline work without waiting for turn completion.

#### Acceptance Criteria

1. WHILE the agent is streaming or executing tool calls, WHEN the user submits a non-empty message via Enter, THE Pie_Cpp_Binary SHALL append the message to the tail of the steering queue and emit a `queue_update` event whose payload includes the current steering queue length and follow-up queue length.
2. WHILE the agent is streaming or executing tool calls, WHEN the user submits a non-empty message via Alt+Enter, THE Pie_Cpp_Binary SHALL append the message to the tail of the follow-up queue and emit a `queue_update` event whose payload includes the current steering queue length and follow-up queue length.
3. WHEN the current assistant turn finishes executing its tool calls and `steeringMode` is `"one-at-a-time"` and the steering queue is non-empty, THE Pie_Cpp_Binary SHALL remove exactly one message from the head of the steering queue, inject it as the next user turn before the next LLM call, and emit a `queue_update` event reflecting the updated queue lengths.
4. WHEN the current assistant turn finishes executing its tool calls and `steeringMode` is `"all"` and the steering queue is non-empty, THE Pie_Cpp_Binary SHALL remove every queued steering message in FIFO order, inject them as the next user turn before the next LLM call, and emit a `queue_update` event reflecting the updated queue lengths.
5. WHEN the agent has no tool calls pending and the steering queue is empty and `followUpMode` is `"one-at-a-time"` and the follow-up queue is non-empty, THE Pie_Cpp_Binary SHALL remove exactly one message from the head of the follow-up queue, inject it as the next user turn, resume the agent loop with that turn, and emit a `queue_update` event reflecting the updated queue lengths.
6. WHEN the agent has no tool calls pending and the steering queue is empty and `followUpMode` is `"all"` and the follow-up queue is non-empty, THE Pie_Cpp_Binary SHALL remove every queued follow-up message in FIFO order, inject them as the next user turn, resume the agent loop with that turn, and emit a `queue_update` event reflecting the updated queue lengths.
7. WHEN the user presses Esc during streaming or tool-call execution, THE Pie_Cpp_Binary SHALL abort the current operation, move all queued messages from the steering queue followed by all queued messages from the follow-up queue into the editor in their original FIFO order separated by newline, leave both queues empty, and emit a `queue_update` event with both queue lengths equal to zero.
8. WHEN the user presses Alt+Up while the editor contains no characters other than whitespace and at least one of the steering or follow-up queues is non-empty, THE Pie_Cpp_Binary SHALL move all queued messages from the steering queue followed by all queued messages from the follow-up queue into the editor in their original FIFO order separated by newline, leave both queues empty, and emit a `queue_update` event with both queue lengths equal to zero.
9. **Property:** FOR ALL steering queues `S` and follow-up queues `F` containing distinct messages, the order of delivery SHALL be: every message in `S` is delivered before every message in `F`, and within each queue the order SHALL be FIFO.
10. **Property:** FOR ALL Message_Queue_Modes `m`, queueing then delivering all messages SHALL preserve message contents byte-for-byte.
11. IF the user presses Enter or Alt+Enter while the editor contains only whitespace or is empty, THEN THE Pie_Cpp_Binary SHALL NOT append any entry to either queue and SHALL NOT emit a `queue_update` event.
12. WHEN the current assistant turn finishes executing its tool calls and the steering queue is empty, THE Pie_Cpp_Binary SHALL proceed to the next LLM call without injecting any user turn from the steering queue.
13. WHEN the agent has no tool calls pending and both the steering queue and the follow-up queue are empty, THE Pie_Cpp_Binary SHALL terminate the agent loop and return control to the editor without injecting any user turn.

### Requirement 12: Bash Execution Semantics

**User Story:** As a user, I want `bash` tool calls and `!command` / `!!command` editor inputs to behave identically to Pi_TS, including timeouts, cancellation, output capture, and truncation, so that command behavior is preserved.

#### Acceptance Criteria

1. WHEN a `bash` tool call, `!command`, or `!!command` is executed, THE Pie_Cpp bash subsystem SHALL spawn a shell using the configured `shellPath` (or the platform default) with the configured `shellCommandPrefix` prepended verbatim to the command, inheriting the current process environment and cwd.
2. THE Pie_Cpp bash subsystem SHALL capture stdout and stderr together as a single byte stream, preserving order byte-for-byte equivalent to the order Pi_TS captures for the same shell invocation.
3. WHEN command output exceeds the truncation threshold used by Pi_TS, THE Pie_Cpp bash subsystem SHALL truncate the in-context output to the same threshold Pi_TS applies, write the full output to a temporary file in the same temporary directory Pi_TS uses, and set `truncated: true` and `fullOutputPath` on the resulting `bashExecution` Session_Entry.
4. IF writing the full-output temporary file fails (disk space, permission, or I/O error), THEN THE Pie_Cpp bash subsystem SHALL set `truncated: true`, omit `fullOutputPath`, include in the `output` field a notice that the full output could not be persisted, and continue without aborting the agent loop.
5. WHEN the user presses Esc during bash execution, THE Pie_Cpp bash subsystem SHALL signal the running process for graceful cancellation, wait for up to the same grace period Pi_TS applies, escalate to a forced kill if the process has not exited, and produce a `bashExecution` Session_Entry with `cancelled: true` and the captured output up to the cancellation point.
6. THE Pie_Cpp bash subsystem SHALL produce `bashExecution` Session_Entries containing the fields `command` (string, the original command text), `output` (string), `exitCode` (integer or the same sentinel Pi_TS uses when the process was cancelled or killed), `cancelled` (boolean), `truncated` (boolean), `fullOutputPath` (string, present only when `truncated: true` and the full output was successfully persisted), and `excludeFromContext` (boolean, set to `true` only when invoked via `!!`, otherwise omitted or `false`).
7. WHEN a `bashExecution` Session_Entry has `excludeFromContext: true`, THE Pie_Cpp Build_Session_Context SHALL omit it from the LLM context while still persisting the entry to the Session_File.
8. THE Pie_Cpp bash subsystem SHALL escape user-provided command tokens such that, for the same input command bytes, the resulting shell invocation is byte-equivalent to the invocation Pi_TS constructs and the captured output is byte-equivalent under the property in criterion 2 (verified for the metacharacters `$`, `` ` ``, `\`, `"`, `'`, `;`, `&`, `|`, `<`, `>`, `(`, `)`, newline, and tab).
9. THE Pie_Cpp bash subsystem SHALL apply the same per-command timeout default Pi_TS applies; on timeout the subsystem SHALL apply the cancellation procedure from criterion 5 and SHALL produce a `bashExecution` Session_Entry with `cancelled: true` and a one-line diagnostic in `output` indicating timeout.
10. IF the shell process cannot be spawned (file not found, permission denied, fork error), THEN THE Pie_Cpp bash subsystem SHALL produce a `bashExecution` Session_Entry with `exitCode` set to the same sentinel Pi_TS uses for spawn failure, `cancelled: false`, and a one-line diagnostic in `output` identifying the cause.


### Requirement 13: Context Files, System Prompt, and Custom Prompt Templates

IF a referenced `<cwd>/.pie/SYSTEM.md`, `<Agent_Dir>/SYSTEM.md`, `<cwd>/.pie/APPEND_SYSTEM.md`, `<Agent_Dir>/APPEND_SYSTEM.md`, `AGENTS.md`, `CLAUDE.md`, or prompt-template file exists but cannot be read (permission error, I/O error, or invalid UTF-8), THEN THE Pie_Cpp_Binary SHALL emit a diagnostic naming the file path and the failure reason, SHALL skip that file, and SHALL continue startup.

### Requirement 14: Skills

**User Story:** As a user, I want skills (Agent Skills standard) to be discovered and invokable as `/skill:<name>` exactly as Pi_TS does, so that on-demand capabilities work the same way.

#### Acceptance Criteria

1. THE Pie_Cpp skill subsystem SHALL discover skills from the following sources: (a) `<Agent_Dir>/skills/`, (b) `~/.agents/skills/`, (c) `<cwd>/.pie/skills/`, (d) `.agents/skills/` walked up from `<cwd>` to the git repo root or filesystem root, (e) skill directories declared by Pie_Packages, (f) the `skills` array in the merged Settings_Layer, and (g) explicit `--skill <path>` CLI arguments.
2. THE Pie_Cpp skill subsystem SHALL recognize as a skill any directory containing a `SKILL.md` file conforming to the Agent Skills standard, plus any top-level `.md` file located directly under `<Agent_Dir>/skills/` or `<cwd>/.pie/skills/` (with the file name as the skill name); deeper directory structures within a skill directory SHALL be treated as skill assets, not as additional skills.
3. IF a discovered skill's `SKILL.md` (or top-level `.md`) lacks a `description` frontmatter key, THEN THE Pie_Cpp skill subsystem SHALL skip that skill (not register it for invocation or auto-discovery) and emit a diagnostic identifying the file path and the missing key.

5. THE Pie_Cpp skill subsystem SHALL inject only the skill's name and `description` (extracted from frontmatter) into the system prompt for progressive disclosure, NOT the full `SKILL.md` body, except for skills whose frontmatter contains `disable-model-invocation: true` which SHALL be omitted from the system prompt entirely while remaining invokable as `/skill:<name>`.

WHEN the user types `/skill:<name> [args]` in the editor and a skill named `<name>` is registered, THE Pie_Cpp_Binary SHALL inject the full `SKILL.md` body as a custom message into the next user turn, append `User: <args>` (where `<args>` is the trimmed remainder of the line) when `<args>` is non-empty, and resume the agent loop.
7. IF the user types `/skill:<name>` for a skill name that is not registered, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic naming the unknown skill, SHALL NOT inject any message, and SHALL leave the editor buffer unchanged.
8. WHERE `enableSkillCommands` in the merged Settings_Layer is `false`, THE Pie_Cpp_Binary SHALL NOT register `/skill:<name>` autocomplete commands but SHALL still discover skills, inject their name and description into the system prompt per criterion 5, and permit model-initiated invocation.
9. WHEN `--no-skills` is passed, THE Pie_Cpp_Binary SHALL skip skill discovery from sources (a)–(f), but SHALL still load any skill explicitly supplied via `--skill <path>`.

### Requirement 16: Pi Packages and Package Manager Subcommands

**User Story:** As a user, I want `pie install`, `pie remove`, `pie update`, `pie list`, and `pie config` to manage extensions, skills, prompts, and themes from npm, git, https, and ssh sources just like Pi_TS, so that my package workflow continues to work.

#### Acceptance Criteria

1. THE Pie_Cpp package subsystem SHALL accept package sources, up to 2048 characters in length, in the formats `npm:<pkg>`, `npm:<pkg>@<version>`, `git:<host>/<user>/<repo>[@<ref>]`, `git:git@<host>:<user>/<repo>[@<ref>]`, `https://<url>[@<ref>]`, and `ssh://<url>[@<ref>]`; IF a supplied source does not match any of these formats or exceeds 2048 characters, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic identifying the rejected source, perform no filesystem or network operations, and exit with a non-zero exit code.

WHEN `pie install <source>` is invoked without `-l` and the source is well-formed, THE Pie_Cpp_Binary SHALL install the package globally to `<Agent_Dir>/git/` (for git/https/ssh sources) or `<Agent_Dir>/npm/` (for npm sources), and on successful completion SHALL exit with code 0; IF the install fails due to network error, source-not-found, or extraction error, THEN THE Pie_Cpp_Binary SHALL leave no partially-installed package directory at the target path, emit a diagnostic indicating the failure cause, and exit with a non-zero exit code.

4. WHEN `pie remove <source>` or `pie uninstall <source>` is invoked and the corresponding installed package directory exists in either the global or project-local location, THE Pie_Cpp_Binary SHALL remove that directory and exit with code 0; IF no installed package directory matches `<source>` in either location, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic indicating the package is not installed, leave the filesystem unchanged, and exit with a non-zero exit code.
5. WHEN `pie update` is invoked, THE Pie_Cpp_Binary SHALL attempt to update each installed package whose source does NOT include a pinned `@<ref>`, leave pinned packages unchanged, and IF an individual package update fails, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic identifying that package, continue updating the remaining unpinned packages, and exit with a non-zero exit code if any update failed (exiting with code 0 only when every attempted update succeeded).
6. WHEN `pie update <source>` is invoked and `<source>` matches an installed package, THE Pie_Cpp_Binary SHALL update that package even if pinned and exit with code 0 on success; IF `<source>` does not match an installed package, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic indicating the package is not installed and exit with a non-zero exit code.
7. WHEN `pie list` is invoked, THE Pie_Cpp_Binary SHALL print to stdout the installed packages from `<Agent_Dir>/git/`, `<Agent_Dir>/npm/`, `<cwd>/.pie/git/`, and `<cwd>/.pie/npm/` in the same textual format Pi_TS produces for the equivalent invocation, and exit with code 0.
8. WHEN `pie config` is invoked and at least one Pie_Package is installed, THE Pie_Cpp_Binary SHALL open an interactive selector that lists every extension, skill, prompt template, and theme contributed by installed packages and lets the user toggle each as enabled or disabled, persisting the resulting enabled/disabled state to the same settings layer Pi_TS uses for this purpose; IF no Pie_Package is installed when `pie config` is invoked, THEN THE Pie_Cpp_Binary SHALL emit a diagnostic stating no packages are installed and exit with a non-zero exit code.
9. THE Pie_Cpp package subsystem SHALL parse the `pi` key in a package's `package.json` (the same manifest key defined by Pi_TS, retained for cross-compatibility with existing packages) to determine which directories to load as extensions, skills, prompt templates, and themes, and IF the `pi` key is present, THEN THE Pie_Cpp package subsystem SHALL use only the directories listed in that key and SHALL NOT additionally auto-discover from conventional directories; IF the `pi` key is absent, THEN THE Pie_Cpp package subsystem SHALL auto-discover from the conventional directories `extensions/`, `skills/`, `prompts/`, and `themes/` at the package root, treating any of those directories that do not exist as empty.
10. WHERE the user has set `npmCommand` in settings to a non-empty argv list, THE Pie_Cpp package subsystem SHALL invoke that command argv to perform npm-based install/uninstall/update operations, using plain `install` (not `install --omit=dev`) as the install verb when `npmCommand` is set.
11. IF a package source resolves to a URL or shell command requiring network access and `PIE_OFFLINE` is set to a truthy value or `--offline` was passed, THEN THE Pie_Cpp_Binary SHALL refuse to perform the network operation, emit a diagnostic stating offline mode is active and identifying the blocked operation, leave the filesystem unchanged, and exit with a non-zero exit code.

### Requirement 20: Image Handling

**User Story:** As a user pasting or dragging images into Pie_Cpp, I want auto-resizing, EXIF orientation correction, in-terminal rendering, and the same configuration knobs as Pi_TS, so that images flow through identically.

#### Acceptance Criteria

1. WHEN the user pastes an image via Ctrl+V (or Alt+V) from the system clipboard, or drags and drops an image file onto the terminal, THE Pie_Cpp image subsystem SHALL decode the image, detect its MIME type from the set {`image/png`, `image/jpeg`, `image/gif`, `image/webp`}, and attach it to the next outgoing message as an `image` content block carrying base64-encoded `data` and the detected `mimeType`.
2. IF a pasted or dropped image has a MIME type outside {`image/png`, `image/jpeg`, `image/gif`, `image/webp`}, or its decoded byte size exceeds 20 MB, THEN THE Pie_Cpp image subsystem SHALL reject the attachment, leave the outgoing message unchanged, and emit a user-visible message indicating that the image was rejected and the reason (unsupported format or size limit exceeded).
3. WHEN `images.autoResize` is `true` (default) and an attached image has a width or height greater than 2000 pixels, THE Pie_Cpp image subsystem SHALL resize the image so that the larger dimension equals 2000 pixels and the smaller dimension is scaled to preserve the original aspect ratio within ±1 pixel; otherwise the image bytes SHALL be transmitted unchanged.
4. WHEN `images.blockImages` is `true`, THE Pie_Cpp image subsystem SHALL drop all attached images from the outgoing message before it is sent to the LLM and emit a user-visible message indicating that images are blocked.
5. WHEN an attached image is a JPEG containing an EXIF Orientation tag with a value in the range 1 through 8, THE Pie_Cpp image subsystem SHALL apply the corresponding rotation and/or mirroring transformation so that the transmitted pixel data matches the visually intended orientation, and SHALL strip or normalize the Orientation tag to value 1.
6. IF a JPEG image has no EXIF Orientation tag, or its Orientation value is outside the range 1 through 8, THEN THE Pie_Cpp image subsystem SHALL transmit the image without applying any orientation transformation.
7. WHEN `terminal.showImages` is `true`, an image is attached or received in a rendered message, and the active terminal advertises support for the Kitty graphics protocol, iTerm2 inline images, or Sixel (checked in that precedence order), THE Pie_Cpp TUI SHALL render the image inline using the highest-precedence supported protocol at a width of `terminal.imageWidthCells` cells, where `terminal.imageWidthCells` is an integer in the range 1 to 200 (default 60).
8. IF `terminal.showImages` is `true` but the active terminal does not advertise support for any of Kitty graphics, iTerm2 inline images, or Sixel, THEN THE Pie_Cpp TUI SHALL render a textual placeholder indicating the image is present and that inline rendering is unavailable in the current terminal, without aborting message rendering.
9. THE Pie_Cpp image subsystem SHALL implement decoding, EXIF handling, and resizing natively (e.g., via `stb_image` plus `stb_image_resize`, or libvips, or Magick++) without depending on the WASM-based `@silvia-odwyer/photon-node` library used by Pi_TS.

#### Acceptance Criteria

1. WHEN Pie_Cpp loads a Session_File with `Session_Header.version = 1`, THE Pie_Cpp session subsystem SHALL convert the linear message sequence into a tree by assigning each message a generated `id` and setting `parentId` to the `id` of the immediately preceding message in the original sequence (with the first message's `parentId` set to null), such that invoking `buildSessionContext` on the resulting tree yields a message list whose order and content are identical to the original v1 sequence.
2. WHEN Pie_Cpp loads a Session_File with `Session_Header.version = 2`, THE Pie_Cpp session subsystem SHALL rename any message whose role equals `hookMessage` to role `custom`, while preserving every other field on that message (including `id`, `parentId`, `content`, timestamps, and any additional keys) byte-equivalent to the source.
3. WHEN Pie_Cpp persists a session that was loaded from a Session_File with `Session_Header.version` less than 3, THE Pie_Cpp session subsystem SHALL write the file with `Session_Header.version = 3`, the migrated entries from criteria 1 and 2 applied, and all unknown fields encountered in the source file retained in their original locations.
4. IF migration of a v1 or v2 Session_File fails for any reason (malformed structure, unreadable encoding, or version-specific conversion error), THEN THE Pie_Cpp session subsystem SHALL leave the original Session_File on disk unmodified, refrain from writing a v3 file, and surface an error indicating the migration failure and the offending file path.
5. IF Pie_Cpp loads a Session_File with `Session_Header.version` greater than 3 or with a missing/unrecognized version value, THEN THE Pie_Cpp session subsystem SHALL refuse to load the session and surface an error indicating an unsupported session version, without modifying the file.
6. WHEN Pie_Cpp reads `auth.json`, `settings.json`, `models.json`, `keybindings.json`, or any theme file previously written by Pi_TS, THE Pie_Cpp auth and settings subsystems SHALL parse the file according to the TS_Wire_Format schema without altering its on-disk contents.
7. WHEN Pie_Cpp writes updates to `auth.json`, `settings.json`, `models.json`, `keybindings.json`, or any theme file, THE Pie_Cpp auth and settings subsystems SHALL serialize the file in the TS_Wire_Format such that a subsequent Pi_TS read of the file succeeds with semantically equivalent content (all keys and values preserved, including any unknown keys round-tripped from the prior read).
8. WHEN Pie_Cpp starts and `<Agent_Dir>` (default `~/.pie/agent/`) does not yet contain `auth.json`, `settings.json`, `models.json`, `keybindings.json`, or a `themes/` directory, but a sibling Pi_TS configuration directory at `~/.pi/agent/` does contain those files, THE Pie_Cpp_Binary SHALL on first run read the corresponding files from `~/.pi/agent/` (without modifying them), copy them into `<Agent_Dir>`, and from that point onward use `<Agent_Dir>` as the canonical location for reads and writes.
9. WHEN Pie_Cpp starts and `<Session_Dir>` (default `<Agent_Dir>/sessions/`) does not yet contain a per-cwd session sub-directory for the current `<cwd>`, but `~/.pi/agent/sessions/--<cwd-with-slashes-replaced>--/` contains one or more Session_Files written by Pi_TS, THE Pie_Cpp_Binary SHALL on first run treat those Pi_TS Session_Files as discoverable for `--continue` and `--resume` (reading them in place under `~/.pi/agent/sessions/`) and SHALL write any newly-created Session_Files for that cwd under `<Session_Dir>`.

### Requirement 24: Diagnostics, Logging, and Error Handling

**User Story:** As a user diagnosing problems, I want clear diagnostics for parse errors, missing tools, failed extensions, network failures, and migration issues that match the level of detail Pi_TS provides, so that I can troubleshoot effectively.

#### Acceptance Criteria

1. IF any of the following failure conditions occur (malformed Session_File line, missing tool name in the Tool_Allowlist, extension load failure, OAuth failure, network failure during update checks or telemetry, package install failure, theme parse failure, migration failure), THEN THE Pie_Cpp_Binary SHALL emit a diagnostic to stderr including: subsystem name, source artifact absolute path (when applicable), failing item identifier, and a one-line reason; and SHALL NOT terminate the host process unless the failure also matches a fatal-exit condition explicitly named in another requirement.
2. WHEN `--verbose` is passed, THE Pie_Cpp_Binary SHALL write to stderr, before entering the selected mode, an enumeration of: each loaded extension (name + absolute path), each loaded skill (name + source path), each loaded prompt template (name + source path), each loaded theme (name + source path), each loaded context file (absolute path), the Model_Registry contents (provider/id pairs), and the effective Settings_Layer values per layer.
3. WHEN Pie_Cpp_Binary completes its primary mode operation without error, THE Pie_Cpp_Binary SHALL exit with exit code 0.
4. WHEN Pie_Cpp_Binary fails due to a CLI argument error (unrecognized flag, missing required argument, conflicting modes per Requirement 2 criterion 9, or `@<path>` resolution failure per Requirement 2 criterion 16), THE Pie_Cpp_Binary SHALL write a diagnostic to stderr identifying the offending argument and exit with a non-zero exit code reserved for argument errors.
5. WHEN Pie_Cpp_Binary fails due to a runtime error (provider error, IO error, signal interruption, or user-initiated abort), THE Pie_Cpp_Binary SHALL write a diagnostic to stderr identifying the failure category and exit with a non-zero exit code distinct from the argument-error exit code defined in criterion 4.
6. **Error-condition property:** FOR ALL malformed JSON inputs to Session_File parsing, theme parsing, settings parsing, auth parsing, and keybinding parsing, the parser SHALL return an error result naming the offending file, the 1-based line number of the first offending line, and a one-line reason citing the JSON token or structural rule that failed; the parser SHALL NOT raise an unhandled exception, abort, or terminate the process.
7. WHEN an extension throws an unhandled exception during event handling, THE Pie_Cpp extension subsystem SHALL emit an `extension_error` event in Json_Mode and Rpc_Mode whose payload includes the extension name, the hook/callback name where the exception occurred, and a one-line reason; surface a visible non-blocking notification in the TUI containing the same fields; continue running all other extensions; and not terminate the host process.
